# Готовые промпты по фазам

Подставляй описание задачи вместо `{задача}`. При необходимости укажи тип: фича/баг, бэкенд/фронтенд.

---

## Research

```
Фаза Research. Задача: {задача}.
Используй правило phase-research. Выбери релевантные файлы, опиши структуру кода и границы — только факты, без рекомендаций. Итог — документ в docs/research/ (или аналог).
```

---

## Design

```
Фаза Design. Задача: {задача}.
Используй правило phase-design. На основе документа Research создай: C4 (контекст, контейнеры, компоненты), dataflow, sequence-диаграммы, ADR с решениями и рисками. Учти стандарты проекта по слоям и паттернам.
```

---

## Planning

```
Фаза Planning. Задача: {задача}.
Используй правило phase-planning. По утверждённому Design создай код-план: фазы реализации, список файлов и изменений в каждой фазе, порядок и зависимости. Выход — docs/plans/ (или аналог).
```

---

## Implementation (три подфазы по порядку: Code → Tests → Code Review)

### Impl.Code (реализация по плану)

```
Подфаза Implementation: Code. Задача: {задача}.
Используй правило phase-impl-code. Реализуй по плану из docs/plans/ и дизайну. Тесты пишет тестировщик в следующей подфазе. Не меняй архитектуру на лету; при расхождениях с докой тестировщик вернёт тебе на доработку.
```

### Impl.Tests (TDD-oriented: опора на дизайн, можно смотреть в код)

```
Подфаза Implementation: Tests. Задача: {задача}.
Используй правило phase-impl-tests. Пиши тесты, опираясь на дизайн/ADR/план; можешь смотреть в код для контекста. Ожидаемое поведение — из документации. Если код расходится с докой — зафиксируй расхождения и передай разработчику на исправление кода, тесты под код не подстраивай.
```

### Code Review (ревью без правок; не путать с Quality Gate)

```
Подфаза Implementation: Code Review. Задача: {задача}.
Используй правило phase-impl-review. Проверь код и тесты: соответствие ADR/плану, линтеры, что тесты опираются на документацию. Выдай список замечаний; сам код не правишь.
```

### Implementation целиком (все три подфазы)

```
Фаза Implementation. Задача: {задача}.
Используй phase-implementation (обзор), затем по порядку: 1) phase-impl-code — реализация по плану; 2) phase-impl-tests — тесты (TDD-oriented, опора на дизайн, при расхождениях с докой — пин разработчику); 3) phase-impl-review — Code Review, замечания списком.
```

---

## Quality Gate (финальный гейт перед коммитом)

```
Фаза Quality Gate. Задача: {задача}.
Используй правило phase-review. Проверь: линтеры, соответствие дизайну и ADR, тесты, безопасность (валидация, инъекции, утечки). Подготовь коммит с кодом и при необходимости обновлённой архитектурной документацией. Пуш не делай — только коммит для человеческого ревью.
```

---

## Полный цикл (старт с нуля)

```
Новая фича: {задача}. Работаем по методологии.
1) Сейчас — фаза Research: собери факты о кодовой базе, релевантные файлы, итог в docs/research/. Используй phase-research.
После результата перейдём к Design.
```
