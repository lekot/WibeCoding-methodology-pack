# WibeCoding-methodology-pack
Методология вайбкодинга

Проблема: роботы тупые. Склонны путаться, циклиться, галлюцинировать. Чем меньше у модели фактов - тем больше галлюцинаций, чем больше у модели фактов, тем более загружено контекстное окно - снова проблемы.
Решение: Дать модели достаточное количество фактов, не перегружая контекстное окно

Как это сделать: AI-Driven Development должна вестись фазами. Каждый агент опирается на документацию и пишет документацию. Не забивая себя поиском фактов (подготовлены) имеет чистое контекстное окно и не имеет пространства для галлюцинаций. 

# Methodology Pack

Набор правил и промптов для работы AI-агентов по фазовой методологии: Research → Design → Planning → Implementation (Code → Tests → Code Review) → Quality Gate.

## Системное правило для всех проектов

Чтобы **любой проект** использовал эту методологию без копирования правил в репозиторий:

1. Открой **Settings → Rules** → блок **User Rules**.
2. Добавь правило с текстом из файла **`FOR-USER-RULES.md`** (скопируй блок «Текст правила»).

После этого агент во всех чатах будет получать указание читать правила из `methodology-pack\rules\` и следовать порядку фаз. Путь в правиле можно поменять, если папку методологии перенёс.

## Что внутри

| Файл | Когда применяется |
|------|-------------------|
| `methodology-overview.mdc` | Всегда (alwaysApply). Напоминает о фазах и контекстной инженерии. |
| `phase-research.mdc` | Включай вручную или по запросу при старте Research. |
| `phase-design.mdc` | При создании/ревью архитектуры. |
| `phase-planning.mdc` | При составлении плана реализации. |
| `phase-implementation.mdc` | Обзор подфаз Implementation (Code → Tests → Code Review). |
| `phase-impl-code.mdc` | Подфаза 1 Code: реализация по плану и дизайну. |
| `phase-impl-tests.mdc` | Подфаза 2 Tests: TDD-oriented, опора на дизайн; можно смотреть в код; при расхождении с докой — пин разработчику. |
| `phase-impl-review.mdc` | Подфаза 3 Code Review: ревью кода и тестов, без правок (не путать с Quality Gate). |
| `phase-review.mdc` | Фаза Quality Gate — финальный гейт перед коммитом. |

Готовые промпты для вызова фаз — в `PROMPTS.md`.

## Как использовать

1. **Новая фича**: Research → Design → Planning → Implementation (Code → Tests → Code Review) → Quality Gate.
2. **Баг или мелкое изменение**: при необходимости Research → Planning → Implementation → Quality Gate.
3. В чате можно явно писать: «Работаем по фазе Research», «Сейчас фаза Design — создай C4 и ADR».

Методология основана на контекстной инженерии и разделении ролей (код / ревью / тесты) для повышения качества ИИ-генерируемого кода.
